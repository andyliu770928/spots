# TRIPMARK DECK 省錢版 MVP 計畫文件

## 1. 專案目標

TRIPMARK DECK 第一版的核心目標，是把使用者在 IG、FB、短影音、文章等來源看到的旅遊靈感，快速整理成可搜尋、可分類、可回看、可手動組成行程的收藏卡。

這一版不追求全自動，也不追求高度智慧推薦，而是先完成一個能穩定上線、成本低、日後容易擴充的 MVP。

---

## 2. 產品定位

TRIPMARK DECK 第一版定位為：

**旅遊靈感收藏與整理工具**

而不是：

- 全自動旅行資料抓取引擎
- 高成本 AI 行程規劃平台
- 重度地圖即時查詢服務

這樣的定位可以把前期技術複雜度、維運壓力與 API 成本都壓到最低。

---

## 3. 第一版產品範圍

### 必做目標

1. 能新增收藏
2. 能編輯收藏
3. 能檢視單筆收藏完整資訊
4. 能用搜尋與篩選整理收藏
5. 能從收藏中手動挑選地點，組成一份簡單行程草稿

### 第一版先不做

1. 全平台自動解析與補完
2. 自動查附近景點 / 美食
3. 自動地址轉座標與大量地圖 API 查詢
4. 自動排最佳路線
5. 每筆內容都做 AI 摘要與推薦

---

## 4. 使用情境

### 情境一：快速收藏旅遊靈感

使用者看到一則旅遊貼文或文章後，把連結貼進網站，快速建立一張收藏卡，並補上標題、分類、城市、摘要與標籤。

### 情境二：整理旅行靈感

使用者可在收藏首頁依照分類、城市、狀態或關鍵字搜尋，快速篩出有興趣的景點、美食、住宿與靈感。

### 情境三：組成一日行程

使用者從已收藏資料中挑選幾個地點，調整順序，加上備註，儲存為簡單的行程草稿。

---

## 5. 頁面規劃

### 5.1 收藏首頁

**用途**：顯示全部收藏，供使用者搜尋、篩選與瀏覽。

**必備功能**：

- 卡片列表
- 關鍵字搜尋
- 分類篩選
- 城市篩選
- 狀態篩選
- 排序：最新新增 / 最近更新

**卡片顯示欄位**：

- 封面圖
- 標題
- 分類
- 城市 / 行政區
- 簡短摘要
- 標籤
- 狀態

**第一版不做**：

- 卡片上直接嵌入地圖
- 複雜動畫
- 高階多條件搜尋器

---

### 5.2 新增 / 編輯頁

**用途**：新增一筆收藏，或補齊已收藏資料。

**必備欄位**：

- 原始連結
- 標題
- 分類
- 城市
- 行政區
- 地址
- 摘要
- 推薦原因
- 標籤
- 圖片
- 狀態
- 備註

**操作邏輯**：

- 使用者貼上連結
- 系統建立一筆收藏資料
- 若能抓到網頁標題或圖片則自動帶入
- 其餘欄位由使用者手動補齊

**第一版不做**：

- 自動補完所有欄位
- 自動查附近景點
- 自動判斷白天 / 夜晚適合度
- 自動生成摘要與推薦文案

---

### 5.3 地點詳情頁

**用途**：顯示單筆收藏的完整資訊。

**必備顯示區塊**：

- 標題
- 圖片
- 分類 / 地區 / 狀態
- 地址
- 原始連結
- 摘要
- 推薦原因
- 標籤
- 備註
- 建立時間 / 更新時間

**地圖策略**：

第一版優先只顯示地址與「在 Google Maps 開啟」按鈕，不直接嵌入地圖元件，避免地圖 API 成本累積。

---

### 5.4 行程規劃頁

**用途**：從已收藏地點中手動組成簡單行程。

**必備功能**：

- 建立行程草稿
- 從收藏中加入地點
- 調整地點順序
- 為每個地點添加簡單備註
- 儲存行程

**行程顯示欄位**：

- 行程名稱
- 日期（可選）
- 地點清單
- 備註

**第一版不做**：

- 自動路線最佳化
- 自動交通時間估算
- 自動推薦早餐午餐晚餐
- 一鍵生成完整 itinerary

---

## 6. 資料結構設計

### 6.1 places

核心收藏資料表。

**建議欄位**：

- id
- title
- category
- city
- district
- address
- lat
- lng
- source_url
- source_platform
- summary
- why_go
- tags
- status
- notes
- cover_image_url
- created_at
- updated_at

**欄位說明**：

- `title`：收藏標題或地點名稱
- `category`：spot / food / hotel / idea
- `city`：城市
- `district`：行政區
- `address`：地址
- `lat`, `lng`：座標，第一版可為空
- `source_url`：原始貼文或文章網址
- `source_platform`：instagram / facebook / article / youtube / other
- `summary`：簡短介紹
- `why_go`：推薦原因或想去理由
- `tags`：標籤陣列
- `status`：inbox / shortlisted / visited / archived
- `notes`：自由備註
- `cover_image_url`：封面圖網址

**設計原則**：

- 資料可不完整，但必須能先存下來
- 地址與座標都不強制
- 先以單張封面圖為主，降低圖片管理複雜度

---

### 6.2 trips

行程主表。

**建議欄位**：

- id
- title
- trip_date
- city
- notes
- created_at
- updated_at

---

### 6.3 trip_items

行程中的地點清單。

**建議欄位**：

- id
- trip_id
- place_id
- sort_order
- note
- created_at

**設計原則**：

- 一個行程可包含多個收藏地點
- 可自由排序
- 每個地點可加簡短備註

---

## 7. 收藏狀態設計

為了讓收藏資料能逐步整理，建議第一版就加入簡單狀態欄位。

**建議狀態**：

- `inbox`：剛收藏，尚未整理
- `shortlisted`：已挑選，可能會去
- `visited`：已去過
- `archived`：先封存

這個欄位會是首頁篩選與整理感的重要基礎。

---

## 8. 新增資料流程

### 8.1 手動新增流程

1. 使用者貼上來源連結
2. 系統建立一筆收藏
3. 使用者手動填寫標題、分類、城市、摘要等欄位
4. 儲存資料

### 8.2 半自動新增流程

1. 使用者貼上來源連結
2. 系統嘗試抓取：
   - 網頁標題
   - Open Graph 圖片
   - 來源平台
3. 將抓到的資料預填進表單
4. 使用者手動修正與補齊
5. 儲存資料

### 第一版避免事項

- 貼連結後就自動執行地址查詢、附近景點查詢、營業時間補完、AI 摘要等大量流程

## 9. 技術架構

### 前端

- **Next.js**

理由：

- 適合頁面型產品
- 可直接接 API routes 或 server actions
- 部署到 Vercel 很方便

### 後端

- **Next.js API routes / server actions**

理由：

- 先不額外維護獨立後端
- 可減少部署與管理複雜度

### 資料庫

- **Supabase Postgres**

理由：

- 便宜好上手
- 與前端串接快
- 之後可接 Storage 與 Auth

### 圖片儲存

- **Supabase Storage（僅在必要時使用）**

第一版以儲存原始圖片 URL 為主，減少儲存與轉檔成本。

### 部署

- **Vercel Hobby**

理由：

- 前期成本低
- 與 Next.js 整合方便

### 地圖策略

第一版先不嵌地圖元件，僅提供 Google Maps 外部連結，避免地圖 API 成本。

## 10. API 規劃

### places

- `GET /api/places`：查詢收藏列表
- `POST /api/places`：新增收藏
- `GET /api/places/:id`：查詢單筆收藏
- `PATCH /api/places/:id`：更新收藏
- `DELETE /api/places/:id`：刪除收藏

### link preview

- `POST /api/link-preview`
  - 輸入：url
  - 輸出：title, image, platform

### trips

- `GET /api/trips`：查詢行程列表
- `POST /api/trips`：新增行程
- `GET /api/trips/:id`：查詢單筆行程
- `PATCH /api/trips/:id`：更新行程

### trip items

- `POST /api/trips/:id/items`：新增地點到行程
- `PATCH /api/trip-items/:id`：更新排序或備註
- `DELETE /api/trip-items/:id`：刪除行程地點

## 12. 成本控制原則

### 原則一：能手動就先不要自動

第一版優先讓使用者手動補資料，避免外部查詢次數過多。

### 原則二：能存連結就先不要重抓內容

先保存原始來源連結與基本預覽資料，不做重度爬蟲。

### 原則三：能外開地圖就先不要嵌入地圖

先用 Google Maps 外部連結，避免地圖 SDK 與 Places API 費用。

### 原則四：圖片先存網址，不急著上傳與壓縮

除非使用者主動上傳，否則先以原始圖片 URL 為主。

### 原則五：AI 功能先做成按鈕觸發，不做全自動背景執行

未來若加入 AI 摘要或推薦，也應採手動觸發模式，避免成本失控。

---

## 13. 第一版成功標準

若第一版達成以下目標，即可視為 MVP 成功：

1. 使用者能在 1 分鐘內完成一筆收藏
2. 收藏資料可被搜尋與篩選
3. 使用者可從收藏中組成簡單行程
4. 成本維持在低月費或免費方案範圍內
5. 架構足夠簡單，方便後續擴充自動補完與推薦功能

---

## 14. 後續擴充方向

待 MVP 驗證後，可依需求逐步擴充：

1. 自動 link preview 強化
2. 地址與座標補完
3. 附近景點與美食推薦
4. 行程路線與時間建議
5. AI 摘要、標籤與文案補完
6. 多圖管理與封面圖排序
7. 多使用者與分享功能

但以上功能都應在 MVP 穩定、需求明確後再加入。

---

## 15. 結論

TRIPMARK DECK 第一版應聚焦在「低成本、可上線、可整理、可擴充」四個核心原則。

最重要的不是把所有旅遊資料一次抓滿，而是先建立一個讓旅遊靈感能穩定落地、可被整理、可被重用的資料庫型產品。

只要先把收藏、整理、搜尋、簡單排行程這四件事做好，這個產品就已經具備很好的 MVP 基礎。
