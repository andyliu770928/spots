# TRIPMARK DECK (Spots) 專案架構書 - 20260327 版

## 1. 本次架構變更
- 保留右下角 `+` 作為人類手動輸入入口。
- 新增 `POST /api/places/ingest` 作為小夏專用的穩定寫入入口。
- 新增 `POST /api/places/resolve` 作為小夏專用的低成本解析入口。
- 將 place 欄位正規化邏輯抽到共用 util，避免 UI 與 agent 各自用不同規則。

## 2. 新增資料流
### 手動輸入
1. 使用者點首頁右下角 `+`
2. `AddPlaceModal` 顯示表單
3. 前端直接透過 `src/lib/api.ts` 寫入 `places`

### 小夏自動寫入
1. 小夏整理好景點資料
2. 呼叫 `POST /api/places/resolve`
3. 取得候選欄位與缺漏欄位
4. 小夏補資料後再呼叫 `POST /api/places/ingest`
5. 帶上 `x-spots-ingest-secret` 或 `Authorization: Bearer <secret>`
6. Route 先驗證 secret，再正規化 payload
7. Route 直接寫入 Supabase `places`

## 4. Resolve API 規格
### Endpoint
- `POST /api/places/resolve`

### Payload
- 必填：`url`
- 選填：`title`, `summary`, `tags`, `category`

### 回傳
- `place`
  - `title`, `category`, `source_url`, `source_platform`, `city`, `district`, `address`, `summary`, `why_go`, `tags`, `cover_image_url`, `status`
- `confidence`
  - `title`, `address`, `category`
- `missing_fields`
  - 尚未補齊的欄位列表

### 解析範圍
- `Google Maps`：優先從 query 直接還原地址
- `一般網頁`：使用 `og:title`、`og:description`、`og:image`
- `Instagram / Facebook 非影片貼文`：先當一般網頁處理，從 metadata 與文字抽 title / summary / tags

## 5. Ingest API 規格
### Endpoint
- `POST /api/places/ingest`

### Auth
- Header `x-spots-ingest-secret: <secret>`
- 或 `Authorization: Bearer <secret>`
- 伺服器端環境變數：`SPOTS_INGEST_SECRET`

### Payload
- 必填：`title`
- 選填：`source_url`, `source_platform`, `category`, `city`, `district`, `address`, `summary`, `why_go`, `notes`, `tags`, `status`, `cover_image_url`

### 規則
- `source_platform` 若未提供，會依 `source_url` 自動判斷
- `category` 與 `status` 若不合法，會退回預設值
- `tags` 可接受字串陣列或逗號分隔字串

## 6. 環境變數需求
- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SPOTS_INGEST_SECRET`
- `SUPABASE_SERVICE_ROLE_KEY`：可選，但建議在 server route 使用

## 7. 設計理由
- UI 自動化容易受 selector 與版面變動影響。
- Server-side ingest 可讓小夏跳過前端操作，降低 token 與失敗率。
- Resolve / ingest 分離後，比較容易控制 token 與失敗回退。
- 保留手動 modal，讓你仍能快速自行新增或修正資料。
